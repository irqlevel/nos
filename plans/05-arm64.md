# Stage 5 — arm64 (AArch64) support

**Goal:** `nos` boots on arm64 and the hypervisor runs a Linux guest there too,
using AArch64 hardware virtualization (EL2 + stage-2 translation). arm64 is
increasingly the server-CPU direction (AWS Graviton, Ampere Altra, Azure Cobalt,
Google Axion), so this is strategic, not just a port.

**Depends on:** Stage 2 for the reusable hypervisor logic. Can begin **in
parallel** with Stages 3–4 on x86 once Stage 2 works, because most of the safe
logic is arch-independent.

**Demo:** the same `curl` VM-create flow, but the whole stack — `nos` host +
Linux guest — running on an arm64 board (or under QEMU HVF on Apple Silicon).

---

## Why this fits our environment unusually well

The main dev machine is Apple Silicon. Today x86 `nos` runs under **QEMU TCG**
(slow emulation). An **arm64 `nos` can run under QEMU's native HVF
acceleration** — the host CPU *is* arm64 — so the dev loop becomes dramatically
faster, and nested virtualization is available on recent Apple cores. This
partly inverts the earlier "TCG is slow" problem that pushed x86 toward SVM.

## How much actually has to be rewritten

The Stage 1 discipline pays its biggest dividend here. Split by layer:

**Reused almost unchanged (the safe `hv` crate):** vmexit dispatcher structure,
device-emulation state machines (UART, interrupt controller timer logic),
`GuestMemory`, the VM lifecycle, the control-plane glue, the manifest/live-update
logic. Virtualization *concepts* map over; only the mechanism differs.

**Rewritten (a new `hv-arch` backend + arch kernel code):**
- **CPU bring-up & exception model:** AArch64 exception levels (EL0/EL1/EL2/EL3)
  replace rings; the hypervisor runs at **EL2**. This is a substantial kernel-side
  change (boot, MMU, exception vectors, context switch).
- **MMU:** AArch64 long-descriptor page tables (4 KB granule, up to 4 levels)
  replace x86-64's. The `mm` walk is re-implemented; the allocator/VA logic above
  it is reusable.
- **Interrupts:** **GICv3/GICv4** replaces LAPIC/IOAPIC. Non-trivial: GIC is a
  different model (distributor + redistributors + CPU interface, LPIs via an ITS).
- **Timers:** the **ARM generic timer** (`CNTV`/`CNTP`) replaces PIT/HPET/TSC
  calibration; it is architecturally specified and simpler to use than x86 TSC
  calibration.
- **Hypervisor mechanism:** instead of VMCS/EPT, use **stage-2 translation** (the
  guest-physical→host-physical second stage) plus the EL2 system registers
  (`HCR_EL2`, `VTTBR_EL2`, `VTCR_EL2`, etc.). Trapped guest exits arrive as EL2
  exceptions with `ESR_EL2` syndrome info — *cleaner* than x86 exit
  qualifications in many cases. **No VMXON/vmread/vmwrite ceremony**; guest/host
  register banks are switched more directly.
- **Boot protocol / firmware:** boot via **UEFI + ACPI** (SBSA servers) or
  **device tree** (embedded boards). The Linux arm64 boot protocol (`Image`, not
  `bzImage`; `x0` → DTB) replaces the x86 `boot_params` path in the loader.
- **Drivers:** a PL011 UART instead of 8250; PCIe via ECAM. NVMe is
  transport-agnostic and largely reusable; the Realtek/Intel NIC drivers do not
  apply — target virtio-net or a board-specific NIC.

## Realistic assessment of effort

This is the **largest port** of the five stages because it touches the whole
low-level kernel (boot, MMU, interrupts, timers, exceptions), not just the
hypervisor. But the *hypervisor-specific* new work is smaller than Stage 2 was,
because the hard-won safe logic transfers. Two sub-milestones:

1. **`nos` boots on arm64** (EL1 kernel: boot, MMU, GIC, generic timer, PL011,
   PCIe, virtio) — this is effectively a second Stage 0, and the bulk of the
   work.
2. **Hypervisor on arm64** (run at EL2, stage-2 tables, `Image` loader, reuse the
   safe `hv` crate) — comparatively contained once (1) is done.

## Sequencing rationale

Listed last because it is the least urgent and the most expensive kernel-side.
But it is worth starting the *host* port (sub-milestone 1) early and in parallel,
precisely because the faster HVF-accelerated dev loop on Apple Silicon can then
accelerate work on *all* the other stages' guest-side testing.

## What to study

- **KVM/arm64** (`arch/arm64/kvm`) — the reference for EL2 operation and stage-2.
- **kvmtool / crosvm** on arm64 — minimal VMM device models and the `Image`+DTB
  boot flow.
- **Cloud Hypervisor arm64** — a Rust VMM already doing this; closest to the
  target shape.
- **ARM ARM** (Architecture Reference Manual) for EL2, stage-2, GIC, and the
  generic timer.

## Exit criteria

- `nos` boots unattended on an arm64 target (SBSA/UEFI server or a dev board, or
  QEMU HVF on Apple Silicon).
- GICv3 interrupts, generic timer, PL011 console, PCIe + virtio all functional.
- The hypervisor runs at EL2 with stage-2 translation and boots a Linux `Image`
  guest to a shell, reusing the safe `hv` crate unchanged.
- The HTTP control plane works identically to x86 (same API, same demo).
